# 5.3 设计的基本原理

#软件工程/设计/原理 原理：抽象、细化、模块化、信息隐藏和局部化、模块独立性、耦合

## 抽象

#软件工程/设计/原理 抽象：把现实世界中一定事物、状态或过程之间存在着的某些相似的方面(共性) 集中和概括起来，暂时忽略它们之间的差异，即抽出事物的本质特性而暂时不考虑它们的细节

#软件工程/设计/原理 抽象层次：模块化解决方法不同层次的抽象

- 抽象最高层：使用问题环境语言，以概括的方式叙述问题的解
- 抽象较低层：更过程化的方法;面向问题的术语结合面向实现的术语描述问题的解
- 抽象最底层：直接实现的方式说明


软件工程过程的每一步都是对软件解法的抽象层次的一次次精化。

|       | 可行性研究 |                 |
| ----- | :---: | --------------- |
| 抽象最高层 |  需求   | 采用用户易于理解的语言表达   |
| 抽象较低层 |  设计   | 采用规范的文字、图表或算法表达 |
| 抽象最底层 |  实现   | 直接用编码语言进行表达     |

#软件工程/设计/原理 抽象包含：过程抽象，数据抽象

过程抽象：表示为一个命名的指令序列，它具有一个特定的和受限的功能

数据抽象：表示为一个命名的说明数据对象的数据集合

## 细化

#软件工程/设计/原理 细化：程序的体系结构是通过逐步细化处理过程的层次而设计出来的。通过逐步分解对功能的宏观陈述而开发出层次结构，直至最终得出用程序设计语言表达的程序

- 抽象与细化互补
- 求精是细化过程
- 逐步细化是一种<font color="#c0504d">自顶向下</font>的设计策略

每一步细化都隐含着一定的设计策略。

## 模块化

#软件工程/设计/原理 模块：在程序中是数据说明、可执行语句等程序对象的集合，或者是单独命名和编址的元素
- 模块是构成程序的<font color="#c0504d">基本构件</font>

#软件工程/设计/原理 模块 举例：高级语言中的过程、函数、子程序等


#软件工程/设计/原理 模块化：把软件系统划分成若干模块的过程。自顶向下逐步细化
- 每个模块完成一个特定的子功能，所有模块按某种方法组装起来，成为一个整体，完成整个系统所要求的功能


#软件工程/设计/原理 模块化 属性

- 接口：模块的输入和输出
- 功能：描述该模块实现什么功能。
- 逻辑：描述模块内部怎么做及所需要的数据
- 状态：该模块的运行环境和条件，调用和被调用

#软件工程/设计/原理 模块化 特性

- 外部特性：接口、功能、状态
- 内部特性：逻辑


- 模块数目增加时每个模块的规模将减小,需要的成本(工作量)减少;
- 随着模块数目增加，设计模块间接口所需要的工作量也将增加。

## 信息隐藏和局部化

#软件工程/设计/原理 信息隐藏 

信息隐藏原理：设计和确定模块，使得一个模块内包含的信息(过程和数据)对于不需要这些信息的模块来说，是不能访问的。

#软件工程/设计/原理 局部化

局部化:把一些关系密切的软件元素物理地放得彼此靠近
- 测试及以后的软件维护期间需要修改软件，使用信息隐藏原理作为模块化系统设计的标准就会带来极大好处

## 模块独立性


#软件工程/设计/原理 模块独立性：软件系统中的每个模块只具有软件要求的具体子功能，而与软件系统中其他的模块接口是简单的

模块独立性体现了抽象、模块化、信息隐藏和局部化的概念
- 有效的模块化(即具有独立的模块)的软件比较容易开发出来
- 独立的模块比较容易测试和维护

#软件工程/设计/原理 模块独立性 定性标准：内聚和耦合
## 耦合

#软件工程/设计/原理 耦合：一个软件结构内不同模块之间互连程度的度量。取决于模块间接口的复杂程度，进入或访问一个模块的点，以及通过接口的数据。
- 设计时力争做到低耦合。尽量使用数据耦合，少用控制耦合和标记耦合，限制公共耦合的范围，完全不用内容耦合

![[计算机/软件工程/attachments/2025-10-21_085521.svg]]

#软件工程/设计/原理/耦合 耦合分类：无直接耦合、数据耦合、标记耦合、控制耦合、外部耦合、公共耦合、内容耦合
### 无直接耦合

#软件工程/设计/原理/耦合 无直接耦合：如果两个模块之间没有直接关系，即它们之间的联系完全是通过主模块的控制和调用来实现的。独立性最强
![[计算机/软件工程/attachments/Pasted image 20251014095801.png]]
### 数据耦合

#软件工程/设计/原理/耦合 数据耦合：两个模块彼此间通过参数交换信息，而且交换的信息仅仅是数据。独立性较强，易维护，可复用

![[计算机/软件工程/attachments/Pasted image 20251014102521.png]]

### 控制耦合

#软件工程/设计/原理/耦合 控制耦合：调用模块与被调用模块之间传递的信息中有控制信息(标志开关或命令，有时这种控制信息以数据的形式出现)，中等耦合程度
![[计算机/软件工程/attachments/Pasted image 20251014102705.png]]
### 特征耦合
#软件工程/设计/原理/耦合 特征耦合（标记耦合）：调用模块和被调用模块之间存在复杂结构变量(数组、结构、对象)之类的数据联系。难理解、难复用、副作用
![[计算机/软件工程/attachments/Pasted image 20251014103252.png]]
### 外部耦合

#软件工程/设计/原理/耦合 外部耦合：模块都访问同一全局简单变量，而不是同一全局数据结构，而且不是通过参数表传递该全局变量的信息，则称之为外部耦合。与结构化程序设计思想相矛盾;可读性差、查错难、维护难、复用难

```c++
while(global variable == 0){
	if (argument xyz > 25)
		module3();
	else
		module4(); 
}
```
### 公共环境耦合
#软件工程/设计/原理/耦合 公共环境耦合：两个以上模块之间通过公共环境进行数据的存取。共享的通用区、内存的公共覆盖区、任何存储介质上的文件。复杂程度随耦合的模块个数而变化;不易修改、可靠性和适应性无法保证、可读性差

![[计算机/软件工程/attachments/Pasted image 20251014104313.png]]
### 内容耦合
#软件工程/设计/原理/耦合 内容耦合：模块直接访问另一模块的内部信息、模块代码重叠、多入口模块。耦合度最高、病态的模块耦合，应该坚决避免



## 内聚

内聚：衡量一个模块内部各个元素彼此结合的紧密程度。信息隐藏和局部化概念的自然扩展
- 内聚和耦合：模块内的高内聚意味着模块间的松耦合
- 根据数据流图划分出的模块通常都是顺序内聚的模块

内聚分类
- 低内聚：偶然内聚Coincidental cohesion、逻辑内聚Logical cohesion
- 内聚中：时间内聚Temporal cohesion、过程内聚Procedural cohesion
- 高内聚：通信内聚Communicational cohesion、顺序内聚Sequential cohesion、功能内聚Functional cohesion

过程内聚与顺序内聚的区别
- 过程内聚强调加工处理的次序，成份之间不一定有数据传递
- 顺序内聚强调数据的顺序，成份之间一定有数据的传递且按顺序执行



![[计算机/软件工程/attachments/Pasted image 20251021094617.png]]

设计时力求高内聚，有能力通过修改设计提高模块的内聚程度并且降低模块间的耦合程度，从而获得较的模块独立性。

### 偶然内聚

偶然内聚：模块由多个完成不同任务的语句段组成，各语句段之间的联系十分松散或根本没有任何联系
- 降低了可维护性、模块不可重用

![[计算机/软件工程/attachments/Pasted image 20251021092117.png]]

### 逻辑内聚

逻辑内聚:一个模块完成的任务在逻辑上属于相同或相似的一类，执行哪个任务由调用者来选择
- 不易维护、不易测试、不易复用

![[计算机/软件工程/attachments/Pasted image 20251021092255.png]]

### 时间内聚
时间内聚：个模块包含的任务必须在同一段时间内执行。
- 模块内操作间关系很弱，但与其它模块的操作却有很强的关联、重用性差
```
Function initialization(){
	open old master file
	open new master file
	open transaktion file
	open print file
	initialize sales district table
	read first transaction recoro
	read first old master record
}
```

### 过程内聚

过程内聚：一个模块内的处理元素是相关的，而且必须以特定次序执行
- 操作间是弱连接的，重用性不好


![[计算机/软件工程/attachments/Pasted image 20251021092830.png]]

### 通信内聚
通信内聚：模块中所有元素都使用同一个输入数据和(或)产生同一个输出数据
- 缺乏重用性

![[计算机/软件工程/attachments/Pasted image 20251021093419.png]]

### 顺序内聚
顺序内聚:一个模块内的处理元素和同一个功能密切相关而且这些处理必须顺序执行(通常一个处理元素的输出数据作为下一个处理元素的输入数据)。
- 重用性不好

![[计算机/软件工程/attachments/Pasted image 20251021093610.png]]

### 功能内聚
功能内聚:模块内所有处理元素属于一个整体，完成一个单一的功能
- 易维护、独立性好、重用性好、易扩充

![[计算机/软件工程/attachments/Pasted image 20251021094315.png]]
