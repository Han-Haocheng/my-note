# 可行性研究的任务和步骤

可行性研究的目的不是解决问题，而是确定问题<font color="#c0504d">是否值得去解决</font>

可行性研究分析过程:

1) 进一步分析和澄清问题定义
2) 分析员应该导出系统的逻辑模型
3) 探索若干种可供选择的主要解法

## 研究任务

应该从下述几个方面研究<font color="#00b0f0">每种解法的可行性</font>

- <font color="#00b0f0">技术可行性</font>：方案、技术、专家是否成熟、具备?
- <font color="#00b0f0">经济可行性</font>：资金投入、运行投入…成本效益分析
- <font color="#00b0f0">操作可行性</font>：进度安排是否合理?系统解决实际问题的程度?
- 其他(社会可行性)：政治、法律、文化等可行性

### 技术可行性

- 开发风险分析：在给定的约束条件下，判断能否设计和实现所提出的功能和性能技术可行性分析
- 资源分析：是否具备开发项目的相关人员、已有的技术和经验、设备、环境、培训等能力
- 技术分析：现有技术能否实现新系统，技术难点、建议采用技术的先进性

### 经济可行性

- 一次性开发费用的估算：购买硬件、软件费用、人员费用、人员培训、材料消耗
- 年运行费用的估算：维护人员、购买硬件、软件升级、物资和杂项
- 效益估算
	- 收益指系统投入而增加的收入或减少的成本
	- 对管理型软件，除一次性投资外一般会增大运行成本。其效益在，于提高工作效率和准确性
	- 政府网站侧重于社会效益

### 社会可行性

1. 社会政治
2. 法律法规
3. 侵权

目的：破坏社会安定、软件不能在社会上或政治上引起侵权、民族团结、国家统一或其它责任问题

主要问题
- 项目内容是否符合国家法律
- 是否有侵犯知识产权或宗教信仰
- 供需双方对软件产品的成果分享及转让权利

### 操作可行性

- 进度安排是否合理：用户现行管理制度、人员素质、操作方式是否可行
- 系统解决实际问题的程度：是否解决了现有问题

### 复查系统规模和目标

分析员访问关键人员，仔细阅读和分析有关的材料，以便对问题定义阶段书写的关于规模和目标的报告书进一步复查确认，改正含糊或不确切的叙述，清晰地描述对目标系统的一切限制和约束。

目的：确保分析员正在解决的问题确实是要求他解决的问题。

我们认为用户需要的是否真的就是用户想要的?用户的期望?系统愿景?

### 研究目前正在使用的系统

研究老系统的观点、视角(viewpoint)现有系统要完成的工作，就是新系统所必须具备的功能。现有系统存在的缺点，新系统必须加以改进现有系统所不具备的功能，又是用户必须的，则新系统一定要预以增加，现有系统所需要的费用是新系统的一个重要的投资依据

### 导出新系统的高层逻辑模型

优秀的设计应该是

- 从现有的物理系统出发，导出现有系统的逻辑模型
- 再参考现有系统的逻辑模型，设想目标系统的逻辑模型
- 最后根据目标系统的逻辑模型建造新的物理系统

使用数据流图和数据字典定义模型。

### 进一步定义问题

1. 定义系统目标
2. 复查系统目标和规模
3. 研究现有系统
4. 设计新系统
5. 再定义系统目标

重复这个过程，直到提出的新系统逻辑模型复合系统目标。


### 导出和评价供选择的解法
- 分析员从系统的逻辑模型出发，以系统要求为目标,利用在数据流图上划分自动化边界的方法，可以导出若干种较高层次的物理解法供用户比较和选择。
- 针对多种方案，用户应从如下三个方面进行筛选:
	- 从技术角度排除那些不现实的方案
	- 从操作角度去掉那些操作方式或操作过程用户不能接受的方案
	- 从经济角度估算每个可能系统的成本/效益
- 为每个在技术、操作和经济等方面都可行的系统制定实现进度表。

### 导出和评价供选择的解法

向用户提出推荐方案，在推荐方案中应清楚地表明本项目的开发价值，成本/效益分析，推荐这个方案的理由。

### 草拟开发计划

分析员为推荐的方案草拟一份开发计划含

- <font color="#c0504d">工程进度表</font>
- 估计各类开发人员、各种资源需要情况和使用时间
- 估计系统生命周期每个阶段的成本
- 给出需求分析的<font color="#c0504d">详细进度表和成本估计</font>
