## 绪论
### 软件危机产生的原因
与软件本身的特点有关，另一方面也与软件开发和维护的方法有关系。

1. 软件的复杂性
2. 软件的可变性
3. 软件的不可见性
4. 软件开发和维护中许多错误认识和方法的形成可以归结于计算机发展早期软件开发的个体化特点
5. 软件开发技术的落后
6. 生产方式的落后
7. 开发工具落后，生产率提高缓慢。

### 软件工程及其特点，三要素

软件工程：是指导计算机软件开发和维护的一门工程学科

软件工程的特点
- 抽象性(软件是逻辑产品)
- 复杂性(涉及多领域知识)
- 易变性(需求随场景调整)
- 不可见性(无物理形态)

三要素：工具，方法，过程

### 软件生命周期定义，主要阶段

软件生命周期：软件产品从规划、提出、实现、使用、维护到停止使用的过程
- 主要阶段：软件定义、软件开发、运行维护

### 软件模型

#### 瀑布模型

| 优点             | 缺点            |
| -------------- | ------------- |
| 迫使开发人员采用规范的方法  | 需求很难在开始时完全确定  |
| 每个阶段必须提交文档     | 文档驱动，风险大      |
| 每个阶段的产品都必须质量验证 | 灵活性差，一旦修改损失惨重 |
|                | 不支持软件复用和集成技术  |

特点
- 阶段顺序固定：严格按需求、设计、编码、测试、维护顺序进行，阶段间有明确里程碑。
- 文档驱动：各阶段依赖完整文档，适合需求明确、变更少的项目
- 缺乏灵活性：难以应对需求变更，后期修改成本高

适用范围
- 需求稳定、变化小且开发人员能够一次性获取全部需求的项目
- 软件开发人员有丰富经验，对于应用领域非常熟悉
- 软件项目本身的风险很低
- 只适合需求明确的项目

#### 原型模型

| 优点                       | 缺点               |
| ------------------------ | ---------------- |
| 快速构建、容易修改                | 不能建立里程碑，软件质量难以控制 |
| 能够处理模糊需求，容易适  <br>应需求的变化 | 资源规划和软件配置管理较为困难  |
| 用户参与                     | 软件结构缺乏整体性        |
| 适于中小型软件开发                | 不能很好适应大型软件系统的开发  |

特点
- 快速构建：快速开发可运行原型，帮助用户明确需求。
- 迭代完善：通过用户反馈反复修改原型，降低需求不明确的风险。
- 原型可废弃或演进：原型可能作为最终系统基础，也可能仅用于需求验证

#### 增量模型

| **优点**                     | **缺点**                 |
| -------------------------- | ---------------------- |
| 待开发系统能够被分成不同构件，可以分批次交付软件产品 | 要求待开发系统能够被划分成不同的软件产品构件 |
| 以构件为单位进行开发降低了软件开发的风险       |                        |
| 开发顺序灵活                     |                        |
特点
- 分批交付：将系统划分为多个增量，分批次开发、交付。
- 核心功能优先：先完成核心功能增量，再逐步扩展。
- 降低交付风险：用户可早期使用部分功能，项目整体可控性增强

适用范围
- 待开发系统能够被模块化
- 软件产品可以分批次交付
- 软件开发人员对应用领域不熟悉，或一次性开发的难度很大
- 项目管理人员把握全局的水平很高

#### 喷泉模型

| **优点**  | **缺点**     |
| ------- | ---------- |
| 灵活性和适应性 | 需求不明确      |
| 迭代开发    | 可能造成资源浪费   |
| 风险控制    | 需要良好的沟通与协作 |

特点
- 面向对象驱动：以对象为中心，支持重用与扩展。
- 阶段并行迭代：分析、设计、编码等阶段可重叠、回溯，灵活性高
- 无缝过渡：阶段间无明显边界，适合需求易变的系统

#### 螺旋模型

| **优点** | **缺点**   |
| ------ | -------- |
| 风险分析   | 风险分析     |
| 利于软件复用 | 对开发人员要求高 |
| 质量保证   |          |

特点
- 风险驱动：每轮迭代前进行风险分析，决定后续走向。
- 循环迭代：每轮循环包含规划、风险分析、开发、用户评估四阶段
- 综合性强：融合瀑布的系统性与原型的迭代性，适用于大型复杂项目

适用范围：复杂的、高风险的大型软件项目

## 需求分析

### 需求分析阶段的主要任务
1. 问题明确定义：确定对功能、性能、环境、用户界面的一系列需求
2. 导出软件的逻辑模型
3. 编写文档：编写“需求说明书”，编写使用手册，编写确认测试计划，修改完善项目开发计划

### 数据流图的作用，要素；数据字典的作用
#### 数据流图DFD

数据流图DFD：需求分析时，描述系统的数据流和处理过程，直观展示系统功能的一种图
- 要素：外部实体、数据处理、数据存储、数据流

#### 数据字典DD

数据字典：对DFD中的所有元素进行详细定义和说明，是DFD的补充内容，保证数据描述的一致性和完整性

## 概要设计

### 设计基本原理


- 抽象：忽略次要细节，聚焦核心功能
- 细化
- 模块化：将软件拆分为独立的模块，降低复杂度，提高复用性
- 信息隐蔽和局部化：模块内部的实现细节对外部隐藏，仅通过接口交互
- 模块独立性：用内聚性和耦合性衡量，目标是高聚、低耦合

### 模块化

模块：构成程序的基本构件

模块化：模块化就是把程序划分成独立命名且可独立访问的模块，每个模块完成一个子功能，把这些模块集成起来构成一个整体，可以完成指定的功能满足用户的需求
- 基本属性：==功能、逻辑、状态 、接口==
- 特性：外部特性、内部特性
	- 外部特性：模块名、参数表、其中的输入参数和输出参数，以及给程序以至整个系统造成的影响
	- 内部特性：完成其功能的程序代码和仅供该模块内部使用的数据
- 模块数目增加，设计模块间接口所需要的工作量也将增加 

### 抽象
抽象：抽出事物的本质特性而暂时不考虑它们的细节 

抽象层次
- 最高层-需求阶段：使用问题环境语言，以概括的方式叙述问题的解（==易于理解的语言==） 
- 较低层-设计阶段：采用更过程化的方法，在描述问题的解时，面向问题的术语与面向现实的术语结合使用（==规范文字、图表、算法等==）
- 最底层-实现阶段：用直接实现的方式来说明（==编码语言==） 

软件工程过程的每一步都是对软件解法的抽象层次的一次精化
### 自顶向下，逐步细化

①、求精实际上是细化过程 
②、抽象与求精是一对互补的概念 
### 信息隐藏和局部化

①、信息隐藏原理：设计和确定模块，使得模块内包含的信息（过程和数据） 对于不需要这些信息的模块来说，是不能访问的

②、局部化：把一些关系密切的软件元素物理地放的彼此靠近

### 模块独立

①、模块独立的重要性 
A、有效的模块化（即具有独立的模块）的软件比较容易开发出来 
B、独立的模块比较容易测试和维护 
②定性标准度量：耦合、内聚 

B、内聚：衡量一个模块内部各元素彼此结合的紧密程度 
模块内聚分为：功能内聚、顺序内聚、通信内聚、过程内聚、时间内 
聚、逻辑内聚、偶然内聚 
③、设计时力求高内聚、低耦合
### 耦合 
耦合：一个软件结构内不同模块之间互连的程度

模块耦合分为：无直接耦合、数据耦合、控制耦合、特征耦合、公共 
（环境）耦合、内容耦合、外部耦合 

耦合的设计原则 
- 设计时力争做到低耦合 
- 尽量使用数据耦合
- 少用控制耦合和特征耦合
- 限制使用公共耦合的范围
- 完全不用内容耦合

#### 无直接耦合 
无直接耦合：两个模块直接没有直接关系，完全通过主模块的控制和调用来实现 
- 特点：模块独立性最强 
#### 数据耦合 
数据耦合：两个模块通过参数交换信息，而且交换的信息仅仅是数据（非控制参数、 
结构等） 
- 特点：独立性强、易维护、可复用 
#### 控制耦合 
控制耦合：调用模块与被调用模块之间传递的信息中有控制信息 
- 特点：中等程度的耦合 
#### 特征耦合（标记耦合）
特征耦合：调用模块与被调用模块之间存在复杂结构变量 
- 特点：难理解、副作用、难复用
#### 外部耦合

外部耦合：模块都访问同一全局简单变量，而且不通过参数表传递该全局变量的信息 
- 和公共（环境）耦合的区别：前者是简单变量，后者是复杂数据 结构 
- 特点：与结构化程序设计思想相矛盾、可读性差、查错难、维护难、复用难

#### 公共耦合

公共耦合（环境耦合）：两个以上模块之间通过公共环境进行数据的存取 
- 公共环境：共享的通用区、内存的公共覆盖区、任何存储介质上的文件 
- 特点：复杂程度随耦合的模块个数而变化、不易修改、可靠性和适应性无法保证、 可读性差 
#### 内容耦合 

内容耦合：一个模块直接访问另一个模块的内部信息、模块代码重叠、多入口模块 
- 特点：耦合度最高、病态的模块耦合、坚决避免 

### 内聚 
#### 偶然内聚 
①、模块内多个完成不同任务的语句段组成，各语句段之间的联系十分松散或 
根本没有任何联系 
②、降低了可维护性、模块不可重用 
#### 逻辑内聚 
①、一个模块完成的任务在逻辑上属于相同或相似的一类，执行哪个任务由调 
用者来选择 
②、不易维护、测试、重用 
#### 时间内聚 
①、一个模块包含的任务必须在同一段时间内执行 
②、模块内操作间关系很弱，但与其它模块的操作却有很强的关联、重用性差 
#### 过程内聚 
①、一个模块内的处理元素是相关的，而且必须以特定次序执行 
②、操作间是弱连接的，重用性不好 
#### 通信内聚 
①、模块中所有元素都使用同一个输入数据和产生同一个输出数据 
②、缺乏重用性 
#### 顺序内聚 
①、一个模块内的处理元素和同一个功能密切相关，而且这些处理必须顺序执 
行 
②、重用性不好 
#### 功能内聚 
①、模块内所有处理元素属于一个整体，完成一个单一的功能 
②、易维护、独立性好、重用性好、
#### 内聚的设计原则 
（1）、设计时力求高内聚 
（2）、通过修改设计提高模块的内聚程度并且降低模块间的耦合程度，从而获得较高的模块独立性
## 详细设计

### 模块的三个属性

功能：模块所完成的具体任务或实现的特定操作，即模块“做什么”。这是模块对外的接口所承诺的行为

逻辑：模块内部为实现其功能所采用的具体算法、流程和控制逻辑，即模块“怎么做”。它描述了功能的内部实现细节

状态：模块在运行期间所能保存的内部数据（如局部变量）。它反映了模块在多次调用之间可能保持的记忆或历史信息。

### 设计工具：N-S、PAD、判定表、判定树

N-S图:无流程线的结构化流程图，避免goto语句，适合描述简单算法

PAD图:问题分析图，二维树形结构，易读且能自动生成代码

判定表:以表格形式描述多条件组合的逻辑判断，适合复杂的分支场景

判定树:以树形结构展示条件和结果的对应关系，比判定表更直观

## 面向对象分析
### 概念理解：对象、类、消息、继承、多态、封装

对象：系统中具有明确标识和行为的实体，是类的实例，包含属性（状态） 和方法（行为）。

类：对一组具有相同属性、方法和语义的对象的抽象描述，是创建对象的模板或蓝图。

消息：对象之间进行通信和请求服务的机制。一个对象通过调用另一个对象的方法来向其发送消息。

继承：一种层次化的类关系，允许子类（派生类） 自动获得父类（基类） 的全部属性和方法，并可进行扩展或重写。核心是代码重用。

多态：指同一操作（方法） 作用于不同类的对象时，可以有不同的解释和执行结果。通常通过继承+方法重写或接口实现来实现。

封装：将对象的属性（数据） 和方法（操作） 包装成一个独立的单元，并对外隐藏内部实现细节，仅通过公开的接口与外部交互。核心是信息隐藏

### UML图：类图、用例图

类图：描述系统的静态结构，展示系统中的类、接口、属性、方法以及类与类之间的关系。是面向对象设计的核心。

用例图：描述系统的功能需求，从用户（参与者）角度展示系统能提供的服务（用例），以及参与者与用例之间的关系

### 类与类之间四种关系
| **关系** | **用途**                 | **表示法**   | **说明**      |
| ------ | ---------------------- | --------- | ----------- |
| 关联     | 表示类实例(对象)  <br>之间连接的描述 | --------  | 普通关联        |
| 聚合     | 表示聚合对象由部  <br>分对象组成    | ------<>  | 部分指向整体      |
| 组合     | 表示类之间整体和  <br>部分的关系    | -----<//> | 部分指向整体(共存亡) |
| 依赖     | 表示两个模型元素  <br>之间的依附关系  | - - - ->  | 依赖类指向独立类    |
| 泛化     | 表示一般与特殊的  <br>关系，适于继承  | ------\|> | 子类指向父类      |
| 实现     | 表示类与接口之间  <br>的关系      | - - - \|> | 类指向接口       |

==细化==(实现)：当对同一个事物在不同抽象层次上描述时，这些描述具有细化关系。细化用来协调不同阶段模型之间的关系，表示各个开发阶段不同抽象层次的模型之间的相关性，常用于跟踪模型的演变。

==泛化==(继承)：“is-a”关系，子类继承父类的特征，用实线+空心三角箭头表示，箭头指向父类。

==关联==：类之间的结构化连接，表示对象间持久的引用关系（如学生与课程），用实线表示，可标注多重性。

==依赖==：一个类的定义依赖另一个类（如方法参数、局部变量），是临时性的使用关系，用虚线箭头表示，箭头指向被依赖的类。

聚合/组合（整体-部分关系）：“整体-部分关系”，作为关联的特例。“has-a”关系，分为：

==聚合==：部分可独立于整体存在（如汽车与轮胎），用空心菱形+实线箭头表示，菱形指向整体。

==组合==：部分与整体生命周期一致（如公司与部门），用实心菱形+实线箭头表示，菱形指向整体

## 用户界面设计

### 人机界面设计的原则

一般交互：涉及信息显示、数据输入和系统整体控制

1. ==一致性==：界面的各个部分及各个层次，在术语、风格、交互方式、操作步骤等方面应尽可能保持一致。
2. ==使用简便==：人通过界面完成一次与系统的交互，所进行的操作应尽可能少，包括把敲击键盘的次数和单击鼠标的次数减到最少，甚至要减少拖动光标的距离。
3. ==启发性==：提供清晰的指示和帮助信息，能够启发和引导用户正确、有效地进行界面操作。
4. ==减少重复的输入==
5. ==减少人脑记忆的负担==
6. ==容错性==：对用户的误操作有容忍能力或补救措施。
7. ==及时反馈==：对那些需要较长的系统执行时间才能完成的用户命令，不要等系统执行完毕时才给出

## 软件实现
### 程序设计风格建议

源程序文档化
- 标识符应当尽量具有实际意义
- 程序应当加注释（序言注释和解释性注释，包含数据说明，语句构造，输入和输出，效率）

数据说明
- 数据说明顺序应当规范，使数据的属性易于查找，从而有利于测试、纠错与维护
- 数据命名要符合望文知义的原则
- 对于复杂的数据结构应加注释，说明在实现时的特点

语句构造
- 应该采用直接清晰构造方式，不要为了追求效率而使代码复杂化


输入输出（与用户应用直接相关）
- 输入输出的格式尽可能统一
- 对输入数据的合法性、有效性进行检查
- 输入操作步骤和输入格式应尽量简单，提示信息明确，易于理解
- 输出清晰简明，有必要的注释说明，尽可能使用格式化图表
- 交互式输入时，提供可用的选择和边界值
- 输入一批数据时，使用数据或文件结束标志，而不要用计数来控制

效率
- 主要为处理机时间和存储器容量
- 效率是性能要求，应该在需求分析阶段确定效率要求；
- 效率是靠好设计方法来提高的；
- 程序的效率和程序的简单程度是一致的，不要牺牲程序的清晰性和可读性来不必要地提高效率。

### 程序设计效率建议

体现在以下几方面。
- 在进行编码以前，应简化算法中的算术表达式和逻辑表达式，使其更加简洁
- 仔细审查嵌套循环，尽量减少循环内部的语句和表达式
- 应尽量避免使用多维数组
- 应尽量避免使用指针和复杂的列表
- 采用效率高的算术运算
- 要避免采用混合数据类型
- 尽可能采用占用内存少的数据类型

## 测试

### 测试目标
就是在软件投入运行之前，对软件的需求分析、设计、实现编码进行最终的审查。
### 测试原则

- 测试用例由输人数据和预期的输出数据两部分组成。需要将程序运行后的结果和预期的输出相比较来测试程序。
- 在输人数据的选择上，不仅要选择合理的输人数据，还要选择不合理的输人数据这样可提高程序运行的可靠性。程序应该对不合理的输人数据给出相应提示。
- 用穷举测试是不可能的。可以通过设计测试用例，充分覆盖所有的条件
- 应该在真正的测试工作开始之前很长时间内，就根据软件的需求和设计来制订测试计划，在测试工作开始后，要严格执行，排除随意性。
- 长期保留测试用例。设计测试用例是一件耗费很大的工作，必须作为文档保存。因为测试不是一次完成的，在测试出错误并修改后，需要继续测试。同时，在以后的维护阶段仍然需要测试。
- 对发现错误较多的程序段，应进行更深人的测试。Pareto原则表明，测试发现的错误中的80%是集中在20%的模块中。发现错误多的程序段质量较差，可能隐藏了更多的错误，同时在修改错误过程中又容易引入新的错误。
- 为了达到最佳测试效果，应该由第三方来构造测试用例，避免程序员测试自己的程序。因为好的测试要求承认程序是有错误的，测试目的是发现错误，所以，程序员的心理是测试自己程序的障碍。

### 白盒测试
白盒测试(结构测试)：基于程序内部逻辑结构的测试，又称

逻辑覆盖（强度由弱到强）：

- 语句覆盖：每条语句至少执行一次
- 判定覆盖：每个判断的真、假分支至少执行一次
- 条件覆盖：每个判断中的每个条件的所有可能取值至少执行一次
- 判定-条件覆盖：同时满足分支覆盖和条件覆盖
- 条件组合覆盖：每个判断中所有条件的取值组合至少执行一次
- 路径覆盖：覆盖程序中所有可能的执行路径（通常难以完全达到）

### 黑盒测试

黑盒测试：基于程序功能规格的测试，不关注内部实现，又称功能测试。


等价类划分：将输入域划分为若干有效等价类（合理输入）和无效等价类（不合理输入）。从每个等价类中选取代表性数据进行测试。

边界值分析：基于“错误常发生在边界附近”的经验，专门测试输入域的边界值及边界附近的值。通常与等价类划分结合使用
